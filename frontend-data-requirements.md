# LifeTrace 当前前端需要 LLM 生成的展示内容

本文档只说明一件事：

> 按照当前 `index.html` 已经存在的页面和 UI，哪些展示内容需要 LLM 生成，然后由前端显示出来。

本文档不包含未来功能，不包含当前前端没有展示位置的字段。

---

## 1. 先明确：哪些东西不需要 LLM

下面这些不应该由 LLM 生成：

```text
某一天有没有数据
某一天是不是缺失
某一天有几条线索
具体时间段
具体地点
具体 App 使用记录
照片时间
OCR 原文
权限状态
收藏状态
跳转目标
底部导航文案
按钮文案
页面固定标题
```

这些应该来自：

```text
本地数据库
Android 原生层
后端规则计算
前端固定 UI
```

LLM 的职责是：

```text
把已经存在的事实线索，总结成用户能读懂的展示文案。
```

---

## 2. 当前前端真正需要 LLM 参与的模块

按当前页面来看，真正需要 LLM 生成文案的地方只有这些：

| 页面 | 当前前端位置 | LLM 生成内容 |
|---|---|---|
| 首页 `homePage` | 今日摘要卡片 | 今日摘要标题、今日摘要正文、摘要标签 |
| 日期页 `calendarPage` | 选中日期摘要卡片 | 摘要标题、摘要正文、关键词 |
| 每日回顾页 `reviewPage` | 今日概览卡片 | 今日概览标题、今日概览正文、标签 |
| 每日回顾页 `reviewPage` | 时间线卡片 | 每个时间段的标题、描述、标签 |
| 搜索页 `searchPage` | 猜你想搜 / 搜索发现 | 搜索建议词、发现入口文案 |
| 搜索页 `searchPage` | 搜索结果卡片 | 结果标题、结果摘要、展示标签 |
| 总结页 `summaryPage` | 本周总结 / 周度总结 | 周概览、主要主线、适合回看、每日概述、相关片段 |
| 月度总结弹层 `summaryPanel` | 月度总结详情 | 月度概览、生活节奏、地点总结、洞察、推荐理由 |

当前不需要 LLM 返回的内容：

```text
memoryQuestions
relatedQuestions
单独的 AI 搜索回答卡片
单独的匹配原因模块
权限页内容
首页顶部固定标题和按钮文案
```

## 3. 首页：今日摘要卡片

### 3.1 当前前端位置

页面：`homePage`

当前首页有三块内容：

```text
首页 hero：Memory rebuilt from your traces / 今天，想看什么？
快速进入：选择日期、今日回看、记忆搜索、近期总结
今日摘要：一张摘要卡片
```

其中首页 hero 和“快速进入”是固定 UI，不需要后端和 LLM 返回。

真正需要后端返回的是“今日摘要”卡片，因为它应该根据用户当天真实数据变化。

当前 UI 中的结构是：

```text
今日摘要

今天的大部分时间，你在学习与整理之间切换
下午在图书馆停留较久，傍晚短暂停留并拍下晚霞；晚上和朋友散步后，手机应用切换明显减少。

图书馆 2h 48m / 晚霞照片 / 晚间散步
```

### 3.2 前端最终需要的数据

```json
{
  "homeTodaySummary": {
    "date": "2026-05-20",
    "headline": "今天的大部分时间，你在学习与整理之间切换",
    "paragraph": "下午在图书馆停留较久，期间整理 LifeTrace 项目资料。傍晚拍摄校园晚霞，晚间回到宿舍后手机应用切换减少。",
    "chips": ["图书馆 2h 48m", "晚霞照片", "晚间放松"]
  }
}
```

### 3.3 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `headline` | 是 | 首页今日摘要标题 |
| `paragraph` | 是 | 首页今日摘要正文 |
| `chips` | 可以 | 可以由规则提取，也可以由 LLM 从候选标签中选择和润色 |

### 3.4 哪些不由 LLM 生成

| 字段 | 来源 |
|---|---|
| `date` | 本地日期或后端规则 |
| 图书馆停留时长 | 位置数据统计 |
| 是否有晚霞照片 | 照片时间、地点、识别结果 |
| 是否晚间放松 | App 使用、位置或行为规则推断 |

### 3.5 和每日回顾页的区别

首页的“今日摘要”是更短的入口文案，目的是让用户一眼知道今天有什么值得回看。

每日回顾页的“今日概览”和“时间线”更详细，适合用户点进去后完整查看。

建议长度：

```text
首页今日摘要标题：15-30 字
首页今日摘要正文：60-120 字
首页 chips：2-4 个
```

### 3.6 LLM 输入应该是什么

LLM 不应该直接猜“今天发生了什么”，而应该拿到后端整理好的事实摘要，例如：

```json
{
  "date": "2026-05-20",
  "facts": [
    "13:37-17:05 图书馆附近停留约 208 分钟",
    "15:12-16:05 使用 WPS 整理 LifeTrace 项目资料",
    "18:34 拍摄校园晚霞照片",
    "20:30 后手机应用切换频率下降"
  ],
  "candidateChips": ["图书馆 2h 48m", "LifeTrace 项目", "晚霞照片", "晚间放松"]
}
```

LLM 只负责把这些 facts 改写成首页能展示的短摘要。

## 4. 日期页：选中日期摘要卡片

### 4.1 当前前端位置

页面：`calendarPage`

当前 UI 中，选中某一天后，会出现摘要卡片：

```text
已选择 5 月 20 日       4 条线索

学习、项目与晚霞
2026年5月20日（周三）是一个以课程学习和项目整理为主线的一天……
```

### 4.2 前端最终需要的数据

```json
{
  "selectedDaySummary": {
    "headline": "学习、项目与晚霞",
    "abstract": "这一天以课程学习和项目整理为主线。上午在教学楼学习课程内容，下午在图书馆整理 LifeTrace 项目资料，傍晚拍摄校园晚霞。",
    "keywords": ["课程学习", "图书馆", "项目整理", "校园晚霞"]
  }
}
```

### 4.3 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `headline` | 是 | 摘要卡片标题 |
| `abstract` | 是 | 摘要卡片正文 |
| `keywords` | 可以 | 可以由规则先提取，再让 LLM 选择或润色 |

### 4.4 哪些不由 LLM 生成

```text
选中日期
线索数量
是否有数据
是否线索不足
```

这些来自数据库和规则。

### 4.5 LLM 输入应该是什么

LLM 不应该只拿到一个日期，而应该拿到当天事实线索，例如：

```json
{
  "date": "2026-05-20",
  "facts": [
    "08:35-11:45 在教学楼附近，有课程表记录",
    "13:37-17:05 在图书馆附近停留约 208 分钟",
    "15:12-16:05 使用 WPS 整理项目文档",
    "18:34 拍摄校园晚霞照片"
  ]
}
```

如果 facts 里没有“数据库系统”，LLM 就不能写“学习数据库系统”。

页面：`reviewPage`

```text
今日概览
今天的大部分时间，你在教学楼和图书馆之间切换……
```

| 字段 | 是否由 LLM 生成 | 说明 |

## 6. 每日回顾页：时间线卡片

### 6.1 当前前端位置

页面：`reviewPage`

当前时间线卡片展示结构是：

```text
时间
标题
描述
标签
```

所以最终给前端的时间线数据**必须包含时间**。

### 6.2 为什么之前文档里有 `segmentId`

`segmentId` 不是给用户看的。

它的作用是：

```text
后端规则层先把一天切成多个真实片段，
每个片段有一个稳定 ID，
LLM 只给这个片段生成标题和描述，
后端再用 segmentId 把 LLM 文案合并回原始时间段。
```

例如：

```text
规则层片段：
segmentId = seg_library_project
timeRange = 13:37-17:05
facts = 图书馆停留、WPS、LifeTrace 项目资料

LLM 返回：
segmentId = seg_library_project
title = 下午在图书馆整理项目资料
description = ...

后端合并后给前端：
timeRange + title + description + tags + evidence
```

前端真正展示的是合并后的结果，不是单独展示 `segmentId`。

### 6.3 前端最终需要的数据

```json
{
  "timeline": [
    {
      "id": "seg_library_project",
      "timeRange": "13:37-17:05",
      "title": "下午在图书馆整理项目资料",
      "description": "下午在图书馆停留较长时间，期间整理 LifeTrace 项目资料，是当天比较集中的项目推进片段。",
      "tags": ["图书馆", "LifeTrace", "WPS"],
      "sourceTypes": ["location", "app_usage", "photo"],
      "evidence": [
        "13:37-17:05 图书馆附近停留约 208 分钟",
        "15:12-16:05 使用 WPS 整理项目文档"
      ]
    }
  ]
}
```

### 6.4 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `title` | 是 | 时间线卡片标题 |
| `description` | 是 | 时间线卡片描述 |
| `tags` | 可以 | 规则提取后可由 LLM 选择或润色 |

### 6.5 哪些不由 LLM 生成

| 字段 | 来源 |
|---|---|
| `id` | 后端规则生成 |
| `timeRange` | 本地数据库或原始记录 |
| `sourceTypes` | 后端根据数据源生成 |
| `evidence` | 后端从真实数据中提取 |

### 6.6 LLM 输入示例

```json
{
  "segmentId": "seg_library_project",
  "timeRange": "13:37-17:05",
  "facts": [
    "图书馆附近停留约 208 分钟",
    "使用 WPS 整理项目文档",
    "截图 OCR 出现 LifeTrace 项目资料"
  ]
}
```

### 6.7 LLM 返回示例

```json
{
  "segmentId": "seg_library_project",
  "title": "下午在图书馆整理项目资料",
  "description": "下午在图书馆停留较长时间，期间整理 LifeTrace 项目资料，是当天比较集中的项目推进片段。",
  "tags": ["图书馆", "LifeTrace", "WPS"]
}
```

### 6.8 重要约束

LLM 只能写 facts 里有依据的内容。

例如：

```text
如果 facts 里只有“教学楼附近停留”，没有课程表、OCR、App 记录，
LLM 不能写“学习数据库系统”。
```

只有当输入里明确有：

```text
课程表：数据库课程
或 OCR：数据库系统
或 App/文档标题：数据库相关资料
```

才可以写“数据库课程”或“数据库系统”。

## 7. 搜索页：搜索结果卡片

### 7.1 当前前端位置

页面：`searchPage` / `searchPanel`

当前前端有搜索结果卡片，结构是：

```text
标题 + 最相关徽标
摘要
来源信息网格
```

当前前端没有单独的：

```text
AI 回答卡片
相关追问
独立匹配原因模块
```

所以当前文档不要求 LLM 返回这些字段。

### 7.2 前端最终需要的数据

```json
{
  "searchResults": [
    {
      "id": "daily_2026_05_20_seg_4",
      "title": "下午在图书馆整理项目资料",
      "summary": "这条记录与你搜索的“下午、图书馆、LifeTrace 项目”最接近。记录显示你在图书馆停留较久，并整理了项目文档。",
      "badge": "最相关",
      "sourceItems": [
        {
          "label": "时间",
          "value": "13:37-17:05"
        },
        {
          "label": "来源",
          "value": "位置 / App / 照片"
        }
      ],
      "targetPage": "reviewPage",
      "targetDate": "2026-05-20"
    }
  ]
}
```

### 7.3 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `title` | 可以 | 可以复用记忆片段标题，也可由 LLM 润色 |
| `summary` | 是 | 搜索结果卡片摘要 |
| `badge` | 否 | 一般由排序规则决定，如“最相关” |

### 7.4 哪些不由 LLM 生成

| 字段 | 来源 |
|---|---|
| `id` | 检索系统 |
| `sourceItems[].label` | 前端或后端固定 |
| `sourceItems[].value` | 真实数据 |
| `targetPage` | 后端规则 |
| `targetDate` | 检索结果 |

### 7.5 搜索流程说明

搜索页不是让 LLM 从零找数据。

推荐流程：

```text
1. 本地/后端先检索候选记忆
2. 规则层确定结果排序、时间、来源、跳转目标
3. LLM 只生成搜索结果卡片里的 summary，必要时润色 title
4. 前端展示 result-card
```

### 7.6 当前不需要的搜索字段

以下字段当前前端没有展示位置，先不要作为必需字段：

```text
searchAnswer
matchReason
relatedQuestions
selectedResults
```

如果以后重新设计搜索页，加了“AI 总结回答”区域或“继续追问”区域，再补这些字段。

## 8. 搜索页：猜你想搜 / 搜索发现

### 8.1 当前前端位置

页面：`searchPage`

当前搜索弹层打开后，在用户真正搜索之前，会展示三块入口：

```text
猜你想搜
搜索历史
搜索发现
```

其中“搜索历史”应该来自用户本地历史记录，不需要 LLM 生成。

“猜你想搜”和“搜索发现”可以由后端根据近期记忆数据生成，LLM 可以参与把候选线索改写成更自然的搜索词。

### 8.2 前端最终需要的数据

```json
{
  "searchGuide": {
    "suggestedQueries": [
      {
        "text": "晚霞",
        "hot": true
      },
      {
        "text": "图书馆之后去了哪",
        "hot": true
      },
      {
        "text": "最近一次散步",
        "hot": false
      },
      {
        "text": "小组作业是哪天",
        "hot": false
      }
    ],
    "discoveryEntries": [
      "5 月的高光时刻",
      "最近常去地点",
      "有照片的日子",
      "学习状态变化"
    ]
  }
}
```

### 8.3 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `suggestedQueries[].text` | 可以 | 根据近期真实线索改写成用户可能会搜的话 |
| `discoveryEntries[]` | 可以 | 根据统计结果生成入口文案 |

### 8.4 哪些不由 LLM 生成

| 字段 | 来源 |
|---|---|
| `hot` | 后端规则或点击频次 |
| 搜索历史 | 本地存储或用户历史搜索表 |
| 是否可点击 | 前端固定交互 |

### 8.5 生成逻辑

推荐流程：

```text
1. 后端先统计近期高频地点、照片场景、日程主题、App/文档主题
2. 后端形成候选词，例如：图书馆、晚霞、散步、小组作业
3. LLM 只负责把候选词改写成自然搜索表达
4. 前端点击后，把这个 text 直接作为搜索 query
```

LLM 不能凭空生成用户没有相关记录的搜索建议。

## 9. 周度总结：本周总结 / 周度总结

页面：`summaryPage`

当前页面里已经有：

```text
本周总结
周度总结
```

打开“周度总结”后，弹层里当前展示：

```text
周度概览
这一周可以怎样理解
主要主线
适合回看
每日概述
相关片段
```

所以周度总结不是未来功能，它是当前前端已经有的展示位置。

### 9.2 前端最终需要的数据

```json
{
  "weeklySummary": {

    "reviewSuggestion": "可以重点回看周三和周五的学习片段。",

    "dailyOverviews": [

        "tags": ["课程表", "教学楼"]
      },
      {
        "weekday": "周三",
        "date": "2026-05-28",
        "summary": "下午在图书馆区域停留较久，傍晚有一张校园照片。",

```

### 9.3 哪些由 LLM 生成

| 字段 | 是否由 LLM 生成 | 说明 |
|---|---:|---|
| `overview` | 是 | 周度概览正文 |
| `focus` | 是 | 这一周的主要主线 |

### 9.4 哪些不由 LLM 生成

|---|---|

| `dateRange` | 后端按日期计算 |
| `dailyOverviews[].date` | 本地数据库或日期规则 |

| `relatedMemories[].targetQuery` | 后端规则或固定搜索入口 |

### 9.5 LLM 输入应该包括


LLM 负责解释和总结，不能自己决定这一周有哪些天有数据。

## 15. 为了实现记忆搜索，后端还必须保存什么

前面大部分内容是在说“前端要展示什么文案”。

但真正要实现记忆搜索，光有这些展示文案不够。

后端还需要把每天、每个时间片段整理成可以检索的结构化记忆文件或数据库记录。这个结构不是直接给页面展示的，而是给搜索系统使用的。

### 15.1 为什么需要结构化记忆

搜索流程应该是：

```text
1. 后端/本地数据库保存结构化记忆片段
2. 用户输入自然语言 query
3. 后端用关键词检索/向量检索/规则匹配找候选片段
4. 后端把候选片段交给 LLM 生成搜索结果摘要
5. 前端展示搜索结果卡片
```

所以，LLM 返回“搜索结果卡片摘要”只能解决展示层，不能替代记忆库本身。

### 15.2 后端建议保存的结构化记忆片段

```json
{
  "memorySegments": [
    {
      "id": "daily_2026_05_20_seg_4",
      "date": "2026-05-20",
      "timeRange": "13:37-17:05",
      "title": "下午在图书馆整理项目资料",
      "summary": "下午在图书馆停留较长时间，期间整理 LifeTrace 项目资料。",
      "keywords": ["图书馆", "LifeTrace", "项目资料", "WPS"],
      "locations": ["图书馆"],
      "sourceTypes": ["location", "app_usage", "photo"],
      "rawEvidence": [
        "13:37-17:05 图书馆附近停留约 208 分钟",
        "15:12-16:05 使用 WPS 整理项目文档"
      ],
      "searchText": "2026-05-20 下午 图书馆 LifeTrace 项目资料 WPS 整理 文档",
      "targetPage": "reviewPage"
    }
  ]
}
```

### 15.3 哪些字段用于搜索，哪些字段用于展示

| 字段 | 主要用途 | 说明 |
|---|---|---|
| `id` | 搜索和跳转 | 每个记忆片段必须稳定 |
| `date` / `timeRange` | 搜索和展示 | 时间必须来自真实数据 |
| `title` / `summary` | 搜索和展示 | 可由 LLM 生成，但必须基于事实 |
| `keywords` | 搜索 | 可由规则和 LLM 共同生成 |
| `locations` | 搜索过滤 | 来自位置数据 |
| `sourceTypes` | 展示和过滤 | 说明命中了哪些数据源 |
| `rawEvidence` | 解释和防幻觉 | LLM 总结时必须参考 |
| `searchText` | 检索 | 把可搜索信息拼成文本，便于关键词或向量检索 |
| `targetPage` | 前端跳转 | 通常由规则决定 |

### 15.4 结论

现在文档里要求的“后端返回给前端展示的内容”，可以让页面展示得更完整。

但要真正实现记忆搜索，还需要额外建设一个“结构化记忆库”：

```text
每日原始数据 -> 规则清洗 -> 记忆片段 memorySegments -> 建索引 -> 搜索接口 -> LLM 摘要 -> 前端展示
```

这个结构化记忆库可以保存成本地 JSON 文件、SQLite 表、后端数据库，或者向量库。选哪种不影响前端，但会影响搜索能力。

## 16. Prompt 编写重点

Prompt 应该强调：

```text
1. 只根据输入 facts 总结
2. 不判断这一天有没有数据
3. 不编造时间、地点、课程、App、照片、人物
4. 如果 facts 没有说明课程内容，就不要写具体课程名
5. 如果 facts 只有“教学楼停留”，只能写“在教学楼附近活动”
6. 输出固定 JSON
7. 不输出多余解释文字
```

特别重要：

```text
LLM 不能因为看到“教学楼”就自动推断“学习数据库系统”。
LLM 不能因为看到“图书馆”就自动推断“自习”。
LLM 不能因为看到“晚上”就自动推断“和朋友聊天”。
```

除非输入 facts 明确提供这些信息。

## 17. 结论

当前前端需要 LLM 生成的内容，应该非常克制：

```text
首页今日摘要
日期摘要
今日概览
时间线标题/描述/标签
搜索结果卡片摘要
猜你想搜和搜索发现
周度总结文案
月度总结文案
推荐查看理由
```

当前前端不需要 LLM 生成：

```text
记忆问题
相关追问
单独 AI 搜索回答
单独匹配原因模块
权限页
收藏状态
固定 UI 文案
```

时间线最终给前端的数据必须包含真实 `timeRange`，但 `timeRange` 来自数据库或规则，不来自 LLM。
